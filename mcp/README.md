# Связка Graylog и нейросети по MCP (Model Context Protocol)

## План и суть решения

### Что такое MCP

**Model Context Protocol (MCP)** — открытый протокол, который даёт LLM (нейросети) доступ к внешним данным и действиям через единый интерфейс. В нашем случае:

- **Graylog** выступает как **MCP-сервер**: отдаёт инструменты (поиск логов, списки стримов, статус системы и т.д.).
- **Клиент** (Cursor, LM Studio, Claude Desktop и т.п.) подключается к Graylog по MCP и к **Ollama** как к модели. Пользователь пишет запрос на естественном языке → модель решает, какие инструменты вызвать → клиент дергает Graylog MCP → ответы подставляются в контекст модели → модель формирует итоговый ответ (анализ логов, отчёт, выводы).

То есть нейросеть и Graylog связываются **не напрямую**, а через **MCP-клиент**, который умеет и вызывать инструменты Graylog, и отправлять запросы в Ollama.

### Архитектура

```
┌─────────────────────────────────────────────────────────────────┐
│  Ты (пользователь)                                               │
│  "Покажи ошибки за последний час и кратко резюмируй"               │
└───────────────────────────────┬─────────────────────────────────┘
                                 │
                                 ▼
┌─────────────────────────────────────────────────────────────────┐
│  MCP-клиент (Cursor / LM Studio / Claude Desktop)                 │
│  - Отправляет промпт в Ollama                                    │
│  - По решению модели вызывает MCP-инструменты Graylog           │
│  - Подставляет результаты в контекст и снова обращается к модели  │
└───────────┬─────────────────────────────────────┬─────────────────┘
            │                                     │
            ▼                                     ▼
┌───────────────────────┐             ┌───────────────────────────┐
│  Ollama (локальная    │             │  Graylog 7.0+              │
│  модель, напр.        │             │  MCP endpoint: /api/mcp    │
│  local-llm / Vikhr)   │             │  Инструменты: поиск логов, │
│  порт 11434/11435     │             │  стримы, индексы, статус  │
└───────────────────────┘             └───────────────────────────┘
```

### Что нужно сделать (краткий план)

1. **Graylog**: включить MCP, создать пользователя только для чтения, выдать API-токен.
2. **Ollama**: уже поднят (Docker или локально), модель для анализа (например Vikhr или local-llm).
3. **MCP-клиент**: настроить один из клиентов (Cursor или LM Studio) так, чтобы он подключался к Graylog по MCP и использовал Ollama как модель.
4. **Проверка**: задать в чате вопрос по логам и убедиться, что модель вызывает инструменты Graylog и отвечает на основе данных.

## Шаг 1: Включить MCP в Graylog

1. Зайди в веб-интерфейс Graylog: `http://127.0.0.1:9000/`.
2. **System → Configurations → MCP**.
3. Включи **Enable MCP** (On), сохрани конфигурацию.

После этого MCP endpoint доступен по адресу: `http://127.0.0.1:9000/api/mcp` (или твой `GRAYLOG_HTTP_EXTERNAL_URI` + `/api/mcp`).

---

## Шаг 2: Пользователь и API-токен для MCP

Для MCP лучше не использовать учётку администратора. Создай отдельного пользователя с правами только на чтение и выдай ему токен.

1. **System → Users and Teams** → создай пользователя (например `mcp-readonly`) с ролями только на чтение (просмотр логов, стримов, системной информации — без прав на изменение настроек).
2. **System → Users and Teams → Tokens** (или в карточке пользователя) → **Create token** для этого пользователя.
3. Скопируй выданный **API token** (одна длинная строка). Он понадобится для заголовка авторизации.

Формат авторизации для MCP — Basic Auth. В поле логина подставляется **сам токен**, в поле пароля — буквально строка `token` (как в документации Graylog):

```bash
# Подставь свой API token вместо YOUR_API_TOKEN
echo -n "YOUR_API_TOKEN:token" | base64 -w0
```

Полученную строку используй в заголовке: `Authorization: Basic <эта_строка>`.

Удобно сохранить её в переменную или в конфиг MCP (см. ниже).

## Шаг 3: Настройка Cursor (Graylog MCP + Ollama)

Cursor умеет подключать MCP-серверы и при необходимости использовать свою или кастомную модель (в т.ч. Ollama, если настроен endpoint).

### 3.1 Добавить Graylog как MCP-сервер в Cursor

1. В Cursor: **Settings → Cursor Settings → Features → MCP** (или **Developer → Edit Config** / настройки MCP).
2. Открой конфиг MCP (файл с `mcpServers`, например в `~/.cursor/mcp.json` или через интерфейс «Edit mcp.json»).
3. Добавь сервер Graylog по шаблону (подставь свой Base64 вместо `<BASE64_TOKEN>`):

```json
{
  "mcpServers": {
    "graylog": {
      "url": "http://127.0.0.1:9000/api/mcp",
      "headers": {
        "Authorization": "Basic <BASE64_TOKEN>"
      }
    }
  }
}
```

Если Graylog доступен по другому хосту/порту (например через туннель), замени URL на свой, например `https://graylog.example.com/api/mcp`.

4. Сохрани конфиг. Cursor подхватит новый MCP-сервер; в чате должны появиться инструменты Graylog (поиск логов, списки стримов и т.д.).

### 3.2 Использовать Ollama как модель в Cursor

- В настройках Cursor можно указать **Custom model / OpenAI-compatible** endpoint. Если Cursor поддерживает локальные модели, укажи `http://127.0.0.1:11435` (или `11434`, смотря какой порт у тебя у Ollama) и модель, например `local-llm`.
- Тогда в одном чате будут и вызовы Graylog по MCP, и ответы от твоей локальной модели.

Альтернатива: оставить модель по умолчанию в Cursor и использовать только Graylog MCP для доступа к логам; нейросеть в Cursor будет вызывать инструменты Graylog и формировать ответ.

---

## Шаг 4: Альтернатива — LM Studio (Ollama-подобный бэкенд + MCP)

Если хочешь явно использовать локальную модель (Ollama или LM Studio) вместе с Graylog:

1. Установи [LM Studio](https://lmstudio.ai/).
2. В LM Studio загрузи модель или настрой подключение к Ollama (если поддерживается).
3. **Settings → Program → Edit mcp.json** и добавь тот же блок `graylog` с `url` и `headers` (см. выше).
4. В чате LM Studio задавай вопросы по логам — приложение будет вызывать MCP-инструменты Graylog и подставлять ответы в контекст модели.

---

## Шаг 5: Проверка

1. Убедись, что Graylog запущен и MCP включён: в браузере или через `curl` проверь доступность (с авторизацией):
   ```bash
   curl -s -H "Authorization: Basic YOUR_BASE64" http://127.0.0.1:9000/api/mcp
   ```
   (Ответ может быть в формате MCP — главное не 401/403.)

2. В Cursor (или LM Studio) в чате напиши, например:
   - «Покажи список стримов в Graylog»
   - «Какой статус системы Graylog?»
   - «Найди сообщения с ошибками за последний час и кратко резюмируй»

   Модель должна вызвать инструменты Graylog (по списку стримов, системному статусу, поиску сообщений) и ответить на основе данных.

## Инструменты Graylog по MCP (кратко)

После подключения клиент получает инструменты вроде:

| Категория      | Примеры инструментов |
|----------------|----------------------|
| Система        | `mcp_preview_get_system_status`, `mcp_preview_get_current_time` |
| Логи/поиск     | `mcp_preview_list_streams`, `mcp_preview_search_messages`, `mcp_preview_aggregate_messages` |
| Индексы/поля   | `mcp_preview_list_indices`, `mcp_preview_list_fields` |
| Безопасность   | `mcp_preview_list_new_security_events` (если есть Security) |

Полный список см. в [документации Graylog MCP](https://go2docs.graylog.org/current/setting_up_graylog/model_context_protocol__mcp__tools.htm).

## Важно

- MCP в Graylog 7.0 в статусе **beta**: не рекомендуется для продакшена, формат инструментов может меняться.
- Для MCP используй пользователя с минимальными правами (только чтение), не админа.
- Ollama и модель (Vikhr / local-llm) должны быть доступны с той машины, где запущен Cursor (обычно `http://127.0.0.1:11434` или `11435`).
