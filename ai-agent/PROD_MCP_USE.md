# Использование MCP + Graylog в проде: обоснование и сценарии

Документ для чек-листа и согласования: как и для чего используется ИИ с Graylog в производственном контуре.

---

## 1. Оптимизированные операции (рекомендуемые сценарии)

MCP в проде целесообразно использовать для **ограниченных по объёму и предсказуемых по нагрузке** операций.

| Сценарий | Инструмент MCP | Нагрузка на Graylog | Обоснование |
|----------|----------------|---------------------|-------------|
| **Подсчёт ошибок за период** | `aggregate_messages` (count, groupings) | Низкая: агрегация на стороне OpenSearch | Один запрос, компактный ответ, нет выгрузки сырых логов |
| **Топ-N по полю** (типы ошибок, источники, сервисы) | `aggregate_messages` (groupings + limit) | Низкая | Только сводка, без пересылки миллионов строк |
| **Процент ошибок от общего объёма** | Два вызова `aggregate_messages` (всего + по ошибкам) | Низкая | 2 запроса, результат — числа |
| **Примеры логов по запросу** | `search_messages` с **limit ≤ 200** | Средняя, ограниченная | Выборка для контекста, не полный дамп |
| **Список потоков/индексов/источников** | `list_streams`, `list_inputs`, `list_indices` | Минимальная | Метаданные, без сканирования логов |
| **Время сервера** | `get_current_time` | Минимальная | Для корректного range_seconds |

**Итого:** основная нагрузка — агрегации (`aggregate_messages`) и небольшие выборки (`search_messages` с жёстким лимитом). Полная выгрузка «всех логов за день» не предусмотрена.

---

## 2. Как используется ИИ с Graylog

- **Пользователь** задаёт вопрос на естественном языке (чат, внутренний портал).
- **Сервис (Logs AI)** отправляет запрос в Yandex GPT; модель решает, какие инструменты Graylog вызвать и с какими параметрами.
- **Graylog MCP** выполняет только разрешённые вызовы (по списку инструментов и с ограничениями в коде).
- **Ответ** — краткий текст на основе агрегатов и/или выборки (например: «За последние 2 часа зафиксировано 47 ошибок, 12 уникальных типов; топ-3: timeout, 5xx, connection refused»).

Роль ИИ: преобразование запроса в язык запросов Graylog (Lucene, диапазоны, группировки) и формулировка ответа. Данные не покидают ваш контур: запросы идут в ваш Graylog по вашим правилам доступа.

---

## 3. Ограничения в коде (защита от перегрузки)

В агенте заданы жёсткие лимиты:

| Параметр | Значение | Назначение |
|----------|----------|------------|
| `MAX_ITERATIONS` | 10 | Максимум раундов «модель → MCP → модель» на один запрос пользователя |
| `MAX_SEARCH_LIMIT` | 200 | Верхняя граница `limit` в `search_messages` (срезы сверху принудительно) |
| `MAX_TOOL_RESULT_CHARS` | 12 000 | Обрезка ответа инструмента перед отправкой в модель (ограничение контекста) |

Дополнительно: нормализация запросов (исключение ведущих wildcard, чтобы избежать «all shards failed» в OpenSearch) и инструкции модели предпочитать `aggregate_messages` для отчётов за период.

---

## 4. Риски и митигация

| Риск | Митигация |
|------|-----------|
| Рост нагрузки на Graylog/OpenSearch | Агрегации вместо выгрузки; лимит 200 на `search_messages`; не более 10 вызовов MCP на запрос |
| Раздувание контекста и стоимости LLM | Обрезка результата инструмента до 12k символов |
| Некорректные запросы к OpenSearch | Нормализация query (без ведущих wildcard), жёсткий limit |
| Доступ к логам | Только с учётом прав API-токена Graylog (те же потоки/индексы, что и у учётной записи MCP) |

---

## 5. Краткие формулировки для чек-листа

- **Для каких оптимизированных операций используется MCP:** подсчёт и группировки (aggregate_messages), небольшие выборки примеров логов (search_messages с limit ≤ 200), чтение метаданных (streams, inputs, indices). Полная выгрузка логов не выполняется.
- **Как используется ИИ с Graylog:** пользователь задаёт вопрос текстом; ИИ формирует вызовы MCP к Graylog, получает агрегаты/выборку и формулирует ответ. Запросы ограничены по количеству и объёму данных.
- **Обоснование для прода:** снижение времени на типовые вопросы по логам (сводки, топ ошибок, примеры), предсказуемая нагрузка за счёт лимитов и приоритета агрегирующих операций.

При необходимости лимиты (`MAX_SEARCH_LIMIT`, `MAX_TOOL_RESULT_CHARS`, `MAX_ITERATIONS`) можно ужесточить или ослабить по согласованию с эксплуатацией.
